# Maintaining Jennifer

This guide explains how Jennifer's code generation works, how to update it for new Go versions, and how to build and test everything locally.

## Architecture overview

Jennifer has three packages:

| Package | Purpose |
|---------|---------|
| `jen/` | Core library — the public API users import |
| `genjen/` | Generates `jen/generated.go` from metadata in `genjen/data.go` |
| `gennames/` | Generates `jen/hints.go` — a map of stdlib package paths to names |

Two files in `jen/` are **generated, not hand-written**:

- `jen/generated.go` (~56KB) — all builder methods (`Parens`, `Block`, `Call`, `For`, `Switch`, etc.) generated from `genjen/data.go`
- `jen/hints.go` (~19KB) — stdlib package name hints, generated by scanning the Go stdlib installed on your machine

## How Go keywords are mapped

Not all Go keywords live in the same place. There are three categories:

| Category | Where defined | Examples |
|----------|---------------|----------|
| **Keyword tokens** | `genjen/data.go` → `keywords` | `break`, `default`, `func`, `range`, `go`, `defer`, `chan`, ... |
| **Group constructs** | `genjen/data.go` → `groups` | `for`, `if`, `return`, `switch`, `case`, `map`, `struct`, `interface` |
| **File-level** | `jen/file.go` (hardcoded) | `package`, `import` |

Group constructs need special rendering (opening/closing tokens, separators), which is why they aren't plain keywords.

Predeclared identifiers (`any`, `comparable`, `bool`, `int`, `nil`, `true`, `false`, etc.) and built-in functions (`append`, `len`, `make`, `clear`, `min`, `max`, etc.) are also in `genjen/data.go`.

## Updating for a new Go version

When a new Go version is released, check three things:

### 1. New keywords or predeclared identifiers

Compare the [Go spec's keyword list](https://go.dev/ref/spec#Keywords) and [predeclared identifiers](https://go.dev/ref/spec#Predeclared_identifiers) against `genjen/data.go`:

- `keywords` slice — plain keyword tokens
- `groups` slice — keywords that are rendered as groups (with brackets, separators, etc.)
- `identifiers` slice — predeclared types, constants, and zero value (`nil`)

Also check `jen/reserved.go` which lists all 25 keywords for validation.

### 2. New built-in functions

If Go adds a new built-in (like `clear`, `min`, `max` were added in 1.21), add a new entry to the `groups` slice in `genjen/data.go`. Follow the pattern of existing built-ins — each one specifies its opening token (e.g., `"clear("`), closing token (`")"`), separator, and parameter names.

### 3. New stdlib packages

The `jen/hints.go` file maps stdlib package paths to their short names (e.g., `"net/http": "http"`). This is used for automatic import resolution. It must be regenerated whenever new packages are added to the stdlib.

## Building locally

### Prerequisites

Install the Go version you want to target. Jennifer's `go.mod` declares the minimum version.

### Build and test

```sh
# Build everything
go build ./...

# Run all tests
go test ./...

# Run vet
go vet ./...
```

### Regenerating generated files

The `//go:generate` directives in `jennifer.go` reference remote module paths, which may not work if the module path has changed. To regenerate locally:

```sh
# Build the generators
go build -o ./bin/genjen ./genjen
go build -o ./bin/gennames ./gennames

# Regenerate jen/generated.go (from genjen/data.go metadata)
./bin/genjen

# Regenerate jen/hints.go (scans your installed Go stdlib)
./bin/gennames -output "jen/hints.go" -package "jen" -name "standardLibraryHints" -standard -novendor -path "./..."

# Clean up
rm -rf ./bin

# Verify
go build ./...
go test ./...
go vet ./...
```

`genjen` reads `genjen/data.go` and emits `jen/generated.go`. If you changed keywords, identifiers, or groups in `data.go`, you must regenerate.

`gennames` scans the stdlib of whichever Go version is on your `PATH` and emits `jen/hints.go`. Upgrading Go and re-running `gennames` is all you need to pick up new stdlib packages.

### Verify generated files are up to date

After regenerating, `git diff` should show only expected changes. If `jen/generated.go` changed without any edits to `genjen/data.go`, something is wrong.

## Adding support for a new language construct

If a new Go version introduces syntax that doesn't compose well with existing primitives:

1. **Add a new group** in `genjen/data.go` → `groups` slice, specifying name, opening/closing tokens, separator, and parameters.
2. **Regenerate** `jen/generated.go` (see above).
3. **Add example tests** in `jen/examples_test.go` — these serve as both tests and documentation.
4. **Run the full suite**: `go build ./... && go test ./... && go vet ./...`

If the construct composes fine with existing API (like range-over-int did with `For`/`Range`/`Lit`), just add example tests to document the pattern — no code changes needed.

## Checklist for a Go version bump

- [ ] Read the new [Go spec language versions appendix](https://go.dev/ref/spec#Language_versions) for language changes
- [ ] Compare spec keywords and predeclared identifiers against `genjen/data.go`
- [ ] Check for new built-in functions
- [ ] Update `genjen/data.go` if needed, then regenerate `jen/generated.go`
- [ ] Regenerate `jen/hints.go` with the new Go version installed
- [ ] Update `go.mod` version (`go mod edit -go=1.XX`)
- [ ] Add example tests for new language patterns in `jen/examples_test.go`
- [ ] Run `go build ./... && go test ./... && go vet ./...`
